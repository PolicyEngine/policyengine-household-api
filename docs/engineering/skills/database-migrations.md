# Database Migration Guidelines

## Overview

This project uses Alembic for analytics database migrations with SQLAlchemy
models managed by Flask-SQLAlchemy.

**Critical:** migrations are the source of truth for analytics database schema.
All table creation and schema changes must go through Alembic migrations.

## Essential Rules

### 1. Generate Schema Migrations With Autogenerate

When an analytics model changes, agents must:

1. Update the model in `policyengine_household_api/data/models.py`.
2. Generate the migration with:

   ```bash
   uv run alembic revision --autogenerate -m "Description"
   ```

3. Read and verify the generated file in `alembic/versions/`.
4. Apply it with:

   ```bash
   uv run alembic upgrade head
   ```

Do not hand-author schema-change migrations. If a custom data migration is
needed, create the revision through Alembic first and then add the data operation
explicitly.

### Modal Current/Frontier Compatibility

Normal Modal release PRs run against one shared analytics database while
`current` and `frontier` workers may both be active. Schema migrations must
therefore be backward-compatible with the previously active worker. Use
expand/contract sequencing and keep table or column drops out of normal Modal
release PRs.

### 2. Baseline Exception

A baseline migration may be hand-authored only when it describes schema that
already exists in production and is notably easier to represent by hand than by
reverse-engineering through Alembic autogenerate. Document that exception in the
migration name or docstring.

### 3. Verify Generated Migrations

After running `alembic revision --autogenerate`, agents must verify:

- `upgrade()` contains only the intended table, column, index, and constraint
  changes.
- `downgrade()` reverses those changes.
- The migration does not persist household variable values or other request
  payload values.
- Foreign keys, uniqueness constraints, nullable settings, and indexes match the
  SQLAlchemy model.
- Alembic did not confuse a rename with a drop-and-add operation.

Autogenerate can miss some changes or produce operations that need small
adjustments. Those adjustments are acceptable after the migration has been
generated by Alembic and reviewed.

## Common Commands

```bash
# Apply all pending migrations.
uv run alembic upgrade head

# Generate migration from model changes.
uv run alembic revision --autogenerate -m "Add analytics table"

# Check current revision.
uv run alembic current

# Show migration history.
uv run alembic history
```

## Local SQLite Smoke Test

Use a disposable SQLite database when verifying migration mechanics locally:

```bash
ANALYTICS_DATABASE_URL=sqlite:////tmp/household_api_analytics.db \
  uv run alembic upgrade head
```

Remove the temp database and rerun the command when validating a fresh upgrade
path.

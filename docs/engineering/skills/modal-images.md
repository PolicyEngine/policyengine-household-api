# Modal Image Definitions

READ THIS BEFORE CHANGING `projects/modal-api/policyengine_household_modal/images.py`
OR ANY MODAL IMAGE/DEPLOY BEHAVIOR. Two production-adjacent outages in July
2026 came from changes that looked correct locally and at deploy time but
crashed every container at boot.

## The execution model that makes this dangerous

A Modal app module (`worker_app.py`, `gateway_app.py`, `canary_app.py`) is
imported in TWO different worlds:

1. **On the deploy machine** (`modal deploy -m …` in CI): the full repository
   checkout exists, the uv workspace is synced, and every first-party package
   is importable. Image factory calls here define what gets built.
2. **Inside every running container**: Modal re-imports the same module to
   load the function objects. There is NO repository checkout, NO
   `pyproject.toml`, NO uv workspace — only what the image ships. The image
   itself is already built; the module import just needs to succeed.

Anything executed at module scope — including image factory calls inside
`@app.function(image=…)` decorators — runs in BOTH worlds. A factory that
does eager work (subprocesses, filesystem reads, imports of packages the
image does not ship) will deploy cleanly and then **crash-loop every fresh
container**, while draining containers from the previous deploy generation
keep answering health probes. That failure mode is invisible to deploy-time
checks and to manual probes; it surfaces as requests hanging until timeout.

## The rules

- Image factories in `images.py` must be free of eager work. Deploy-machine-
  only logic is guarded with `if not modal.is_local(): return
  modal.Image.debian_slim(...)` — inside containers the returned image object
  is a placeholder; the real image was fixed at deploy time.
- Third-party dependencies come from requirements files
  (`requirements-modal-worker.txt`, `requirements-modal-gateway.txt`)
  generated by `.github/scripts/modal-deploy-release.sh` via
  `uv export --frozen … --package policyengine-household-modal-api` before
  any `modal deploy` runs. Do NOT reintroduce hand-pinned dependency lists
  (they drift from the lockfile) and do NOT call `Image.uv_sync` (Modal's
  documentation states uv workspaces are not supported).
- Imports of the core package (`policyengine_household_api`) may only appear
  lazily inside the worker factory: gateway and canary images do not ship it.
- First-party packages enter images via `add_local_python_source` and must be
  listed explicitly; when a new workspace member becomes a runtime dependency
  of the worker, add it to `FIRST_PARTY_PACKAGES`.
- The deploy environment must be a full `uv sync --all-packages` checkout, or
  `add_local_python_source` silently drops packages it cannot resolve.

## Mandatory validation before merging image/deploy changes

Deploy the affected apps to a throwaway Modal environment and hit their
endpoints — INCLUDING the gateway app, whose container-boot path is the one
that has broken twice:

```bash
uv run modal environment create <throwaway>
uv run modal secret create household-api ANALYTICS__ENABLED=false AUTH__ENABLED=false --env <throwaway>
uv export --frozen --no-dev --no-emit-workspace --no-hashes \
  --package policyengine-household-modal-api -o requirements-modal-gateway.txt
uv export --frozen --no-dev --no-emit-workspace --no-hashes \
  --package policyengine-household-modal-api --extra worker -o requirements-modal-worker.txt
MODAL_ENVIRONMENT=<throwaway> uv run modal deploy --env <throwaway> -m policyengine_household_modal.gateway_app
curl <deployed-gateway-url>/liveness_check   # proves containers BOOT, not just that deploy succeeded
uv run modal environment delete <throwaway> --yes
```

A successful `modal deploy` proves nothing about container boot; only a
served request does.
